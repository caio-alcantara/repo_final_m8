Esta seção detalha a análise dos logs de comunicação entre a aplicação Android Unitree Go e o robô Go2, focando em Bluetooth Low Energy (BLE) e conexões WiFi, com base em arquivos de log capturados.

### 6. Objetivo

O objetivo destes testes é realizar uma auditoria "Caixa Preta" para compreender os mecanismos de comunicação via Bluetooth e WiFi entre a aplicação Android e o robô Go2. O foco é identificar padrões de conexão, protocolos utilizados e limitações nos logs, facilitando a engenharia reversa dos fluxos de dados e comandos.

### 6.1 Metodologia

Os testes foram conduzidos sob os seguintes princípios rigorosos:

* **Análise Passiva de Logs:** Extração e filtragem de logs do sistema Android via ADB, focando em pacotes relacionados a Bluetooth e WiFi.
* **Estímulo Exaustivo:** Execução de comandos de conexão e desconexão para mapear o comportamento completo da comunicação.
* **Isolamento de Ambiente:** Testes realizados em setup controlado, sem interferência externa de rede.
* **Persistência de Dados:** Armazenamento de arquivos de log para análise detalhada, disponíveis no diretório [unitree-app-logs](/home/asvarius/Área de trabalho/2025-2B-T12-EC08-SecDoc/ferramentas/robô/unitree-app-logs).

### 6.2 Diário de Análise

#### Dia 1: Análise Inicial de Logs Bluetooth e Identificação de Protocolos

A abordagem inicial focou na análise de logs Bluetooth para mapear a comunicação BLE.

* **Procedimento:** Filtragem de logs via ADB nos arquivos [logs-G02-bluetooth.logs](/home/asvarius/Área de trabalho/2025-2B-T12-EC08-SecDoc/ferramentas/robô/unitree-app-logs/logs-G02-bluetooth.logs) (3.563 linhas) e [logs-G02-control-bluetooth.logs](/home/asvarius/Área de trabalho/2025-2B-T12-EC08-SecDoc/ferramentas/robô/unitree-app-logs/logs-G02-control-bluetooth.logs) (3.328 linhas), buscando padrões de scan e conexão GATT.
* **Observações:** Identificada a utilização de Bluetooth Low Energy (BLE) com protocolo GATT para conexões diretas ao robô Go2, identificado como "Go2_25156". Componentes como DogBluetoothActivity e BluetoothService gerenciam as conexões, mas não há dados específicos de comandos transmitidos.
* **Conclusão Crítica:** Os logs revelam handshakes de conexão, mas não capturam payloads de aplicação ou comandos de movimento, indicando necessidade de análise mais profunda em camadas de aplicação.

#### Dia 2: Análise de Logs WiFi e Tentativas de Rastreamento de Pacotes

A estratégia evoluiu para análise de conexões WiFi e tentativas de captura de pacotes.

* **Procedimento:** Análise do arquivo [logs-G02-wifi.logs](/home/asvarius/Área de trabalho/2025-2B-T12-EC08-SecDoc/ferramentas/robô/unitree-app-logs/logs-G02-wifi.logs) (2.644 linhas), complementada por verificação em [logs-G02-app-full.logs](/home/asvarius/Área de trabalho/2025-2B-T12-EC08-SecDoc/ferramentas/robô/unitree-app-logs/logs-G02-app-full.logs) (2.164 linhas) e captura de rede via [unitreecaps.pcapng](/home/asvarius/Área de trabalho/2025-2B-T12-EC08-SecDoc/ferramentas/robô/unitree-app-logs/unitreecaps.pcapng) (28MB, 134.623 linhas).
* **Observações:** Conexões WiFi estabelecidas com SSID "InteliRobo" usando WPA2-PSK, com handshakes de autenticação e parâmetros como SNR de 33 dB e throughput estimado de 171 Mbps. Tentativas de rastreamento de pacotes via WifiHAL retornaram zero pacotes, e não há evidência de comandos de movimento nos logs.
* **Conclusão Crítica:** Os logs WiFi focam em gerenciamento de conexão, sem dados de aplicação ou protocolos como UDP/TCP, sugerindo que comandos são transmitidos sobre a conexão estabelecida, exigindo ferramentas como tcpdump para captura.

#### Dia 3: Integração de Análises e Recomendações para Captura Avançada

Com base nas análises anteriores, integrou-se as descobertas de Bluetooth e WiFi para mapear a estrutura completa.

* **Procedimento:** Cruzamento de dados dos logs Bluetooth e WiFi, com ênfase em limitações e propostas de ferramentas para captura mais profunda.
* **Observações:** A comunicação BLE usa GATT para conexões diretas, enquanto WiFi gerencia redes como "InteliRobo" sem expor dados de aplicação. Recomenda-se uso de Wireshark para tráfego BLE/WiFi e filtragem ADB para logs de aplicação.
* **Resultado:** Identificada a ausência de dados de comandos em todos os logs analisados, validando a necessidade de interceptação ativa para engenharia reversa.

**Vulnerabilidades Identificadas neste ciclo:**

1. **Exposição de Protocolos:** Conexões BLE via GATT e WiFi WPA2 sem criptografia de aplicação aparente, suscetíveis a sniffing.
2. **Ausência de Logs Detalhados:** Logs do sistema não capturam payloads, permitindo comunicações ocultas sem rastreamento.
3. **Dependência de Conexão Direta:** Handshakes expostos sem validação robusta, potencial para ataques de replay ou injeção.
4. **Segurança por Obscuridade:** Parâmetros como portas e UUIDs não explicitados nos logs, mas acessíveis via captura externa.

### 7. Conclusões Gerais

A análise integrada dos logs de Bluetooth e WiFi revela que a comunicação entre o aplicativo Unitree Go e o robô Go2 ocorre principalmente via Bluetooth Low Energy (BLE) com protocolo GATT para conexões diretas e WiFi para redes como "InteliRobo", com autenticação WPA2. No entanto, os logs analisados (disponíveis no diretório [unitree-app-logs](/home/asvarius/Área de trabalho/2025-2B-T12-EC08-SecDoc/ferramentas/robô/unitree-app-logs)) focam em gerenciamento de conexões e não capturam dados de aplicação, comandos de movimento ou protocolos como UDP/TCP. Isso indica que os comandos são transmitidos em camadas superiores, exigindo ferramentas de interceptação como Wireshark ou Frida para análise completa.

**Atenção Crítica:** A análise revelou a presença de dados sensíveis em texto claro nos logs, incluindo senhas WiFi, chaves de API e identificadores de dispositivo. Estes dados representam riscos significativos de segurança e devem ser sanitizados antes de qualquer compartilhamento ou publicação dos logs. Recomenda-se alteração imediata de credenciais expostas e implementação de sanitização de logs na aplicação.
